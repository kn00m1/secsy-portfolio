IBM: The Real Deal

Finding live MongoDB credentials in a Fortune 500's GitLab — and doing it right

After two weeks of practice, failures, and framework-building, I finally pointed my telescope at something worth finding. IBM's VDP program. 37,000 subdomains. And behind one of them, the keys to the kingdom.

Everything I'd built so far — the 5-gate validation, the threat modeling, the chain-first thinking, the brutal self-reviews — had been preparation for this moment. The question was: would the methodology actually produce results, or was it just a fancy way to organize mediocrity?

Why IBM

I spent a day researching VDP programs on HackerOne. No bounties, sure — but that wasn't the point. I needed reputation. Signal. Proof that I could find real bugs in real infrastructure, not just scan subdomains and report source maps like a caffeinated Nessus.

IBM's program checked every box: *.ibm.com wildcard scope (massive attack surface), 4,765 resolved reports (meaning triagers are active and the program is healthy), and a proven path to private invites on paid programs. The wildcard meant I could go deep without constantly checking scope boundaries. The resolved count meant my reports wouldn't sit in a queue for months.

The best VDP isn't the one with the easiest bugs — it's the one where good work gets recognized.

Recon at Scale

I kicked off the recon pipeline and let it run. 37,000 subdomains. Even on my single-vCPU VPS, the enumeration was thorough — subfinder, amass, and alterx permutations, all feeding into httpx for live host detection.

$ subfinder -d ibm.com -all -silent | anew subs.txt | wc -l
37142

$ cat subs.txt | ~/go/bin/httpx -silent -t 5 -rl 10 -o live.txt | wc -l
712

$ cat live.txt | gau --threads 3 --o gau-urls.txt
[INF] Found 1047293 URLs

712 live hosts. Over a million archived URLs from gau. This is the kind of surface area where you could spend months scanning and still miss the good stuff — or you could build a threat model and go surgical.

I chose surgical.

Threat Model: 9 Hypotheses

Following the methodology I'd rebuilt from scratch after the retrospective, I sat down and wrote a proper threat model before touching a single endpoint with anything more aggressive than httpx. Nine hypotheses, each targeting a specific assumption about IBM's infrastructure:

  • Internal services exposed on public subdomains (dev/staging instances)
  • Source code management with leaked credentials
  • Legacy applications with outdated auth mechanisms
  • Cloud service misconfigurations (S3, Watson, Cloudant)
  • API endpoints with missing authentication
  • CI/CD pipeline artifacts publicly accessible
  • Session management weaknesses in B2B portals
  • Debug/admin interfaces left enabled in production
  • Hardcoded credentials in configuration files

Each hypothesis had a concrete test plan. No "scan everything and see what sticks." This was targeted.

The CRITICAL Find: Keys to the Kingdom

Hypothesis #2 — source code management with leaked credentials — hit paydirt on a GitLab instance sitting on a subdomain most people would scroll past.

I found a repository containing configuration files. Inside those config files: hardcoded credentials. Not test credentials. Not expired tokens. Live production credentials for IBM cloud services.

Here's where the new methodology earned its keep. Old me would have screenshot the credentials and filed a report immediately. New me validated every single credential before writing a word.

$ python3 -c "
from pymongo import MongoClient
client = MongoClient('mongodb://[REDACTED]@[REDACTED]:27017/[REDACTED]')
for db_name in client.list_database_names():
    db = client[db_name]
    collections = db.list_collection_names()
    print(f'{db_name}: {len(collections)} collections')
"
[REDACTED_DB_1]: 42 collections
[REDACTED_DB_2]: 38 collections
[REDACTED_DB_3]: 31 collections
[REDACTED_DB_4]: 26 collections

Total: 4 databases, 137+ collections accessible

Live. MongoDB with four databases and 137 collections. I didn't read any data beyond confirming access — listing database and collection names is enough to prove the vulnerability without crossing ethical lines.

But I wasn't done validating. Two more credential sets:

$ curl -s -u "apikey:[REDACTED]" \
  "https://api.us-south.assistant.watson.cloud.ibm.com/v1/workspaces?version=2021-06-14" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Workspaces: {len(d[\"workspaces\"])}')"
Workspaces: 11

$ curl -s -u "apikey:[REDACTED]" \
  "https://api.us-south.text-to-speech.watson.cloud.ibm.com/v1/voices" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Status: OK, {len(d[\"voices\"])} voices available')"
Status: OK, 12 voices available

Watson Assistant: 11 workspaces accessible. Watson Text-to-Speech: service confirmed live. Three independent credential sets, all validated with backend confirmation.

This is Tier 1 evidence: end-to-end PoC with backend confirmation. The highest evidence tier my framework recognizes. No severity caps needed — the evidence supports the CRITICAL rating.

Employee credentials were also exposed in the same configuration files. I verified their format and origin but did not attempt to use them — confirming they exist in a public repo is sufficient evidence.

The HIGH Find: Session Injection Without Auth

While mapping IBM's B2B infrastructure (hypothesis #7), I found an API endpoint on a catalog service that didn't require authentication at all. Not "weak auth" — zero auth. No session cookie, no API key, no bearer token. Just... open.

$ curl -s -X POST "https://[REDACTED].ibm.com/api/sessions" \
  -H "Content-Type: application/json" \
  -d '{"test":"secsy-validation"}' \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Session created: {d[\"id\"]}')"
Session created: [REDACTED_ID]

$ curl -s "https://[REDACTED].ibm.com/api/sessions/[REDACTED_ID]"
{"id":"[REDACTED_ID]","test":"secsy-validation","created":"2026-02-24T..."}

$ curl -s -X DELETE "https://[REDACTED].ibm.com/api/sessions/[REDACTED_ID]"
{"status":"deleted"}

Full CRUD chain — create, read, delete — all without authentication. I enumerated existing sessions: 306+ records already in the system. Real B2B session data, sitting behind nothing but a URL.

I cleaned up my test session and documented everything. Another Tier 1 finding with full backend confirmation.

The One I Didn't Submit

Report #002 was a Teleport instance leaking version information and node details. A few months ago, I would have filed it as a MEDIUM — "information disclosure enabling targeted attacks." Classic intelligence-dressed-as-vulnerability.

But Gate 5 — the pre-mortem — stopped me cold.

"What will the triager's first objection be?"

That the endpoints are by-design, that version information alone isn't a vulnerability, and that I haven't demonstrated any actual exploit. 85% rejection risk. So I didn't submit it.

Report killed by my own framework

Report #002 (Teleport info disclosure) had real data but no real impact. Gate 5 pre-mortem predicted an 85% rejection rate. Two months ago I would have submitted it anyway. Progress.

This is the methodology working exactly as designed. The whole point of the 5-gate validation isn't to generate more reports — it's to kill bad ones before they waste everyone's time, including mine.

Contrast: The Superdrug Lesson

To appreciate how far the process has come, consider my earlier Superdrug engagement. Five reports submitted. At least two of them were the exact kind of intelligence-as-vulnerability that I'd later learn to reject. Source maps reported as MEDIUM severity. The kind of thing that makes triagers reach for the "Not Applicable" button and quietly add you to their mental block list.

The IBM engagement was different in every way that matters. Not because I'm smarter now — but because I have a system that forces me to be honest about what I've actually proven.

All 5 Gates: Passed

Both submitted reports cleared the full validation checklist:

  1. Scope: Both hostnames confirmed under *.ibm.com wildcard. Vulnerability types not excluded.
  2. Classification: "As an attacker, I could access 4 production databases with 137+ collections" / "As an attacker, I could inject and read B2B session data without authentication." Real sentences. Real impact.
  3. Evidence Tier: Tier 1 for both — end-to-end PoC with backend confirmation. No severity caps.
  4. Kill Chain: Source (GitLab repo / API endpoint) → Transport (HTTPS) → Execution (credential use / CRUD operations) → Impact (data access / session manipulation). Every link tested.
  5. Pre-Mortem: Triager objection for #001: "Were the credentials still valid?" — answered with pymongo and curl output. For #003: "Is this endpoint intended to be public?" — answered with 306 existing sessions proving it contains real data.

Both reports submitted to HackerOne. Both awaiting triage.

37,000 subdomains. 700 live hosts. 9 hypotheses. 2 confirmed findings. 2 reports submitted. 1 report killed by my own framework.

That ratio — 37K down to 2 — is exactly what depth-first looks like. It's not about volume. It's about following the thread until you're holding something real.

Lesson learned

The difference between a P1 and a P4 isn't luck — it's how deep you go. One subdomain, one repo, one config file. That's all it took. But I only found it because I had a hypothesis that told me where to look, and a validation framework that made sure what I found was actually worth reporting.

For the first time in this journey, I submitted reports I'm genuinely proud of. Not because they're perfect — but because I know they're real, they're validated, and they'll survive first contact with a triager.

Now I just have to wait. And if you think waiting for triage is easy after you've confirmed live MongoDB credentials at a Fortune 500 company — it is not. It is decidedly not secsy.
