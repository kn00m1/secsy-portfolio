Five Targets, One Painful Lesson

How I launched 5 bug bounty engagements and learned scope the hard way

Ready to go from practice to production. What could go wrong? Everything.

February 18th. Three days after the Juice Shop speedrun. I'm feeling invincible. I've got 26 CTF flags, a fully armed VPS, and the unearned confidence of someone who has never had a report rejected. I sit down, crack my knuckles, and make the single worst strategic decision of the entire project:

I sign up for five bug bounty programs simultaneously.

HackerOne. Bugcrowd. Intigriti. Three platforms, five programs, one researcher who doesn't know what the word "scope" means yet. Let me walk you through the carnage.

Platform Onboarding Speed Run

I registered accounts on all three platforms in one sitting. HackerOne as akshayraw, Bugcrowd and Intigriti as secsy. Email verification, profile setup, 2FA, the works. Then I went shopping for programs like a kid in a candy store, picking targets based on vibes instead of strategy:

$ cat ~/engagements/active.txt
1. meesho          - HackerOne   (Indian e-commerce, huge scope!)
2. superdrug       - Intigriti   (UK pharmacy, healthcare data!)
3. capitalcom      - Intigriti   (Trading platform, financial data!)
4. deriv           - HackerOne   (Trading platform, money stuff!)
5. just-eat-takeaway - Bugcrowd  (Food delivery, everyone uses it!)

Total programs: 5
Total experience: 0
Total hubris: immeasurable

Five targets. Five recon pipelines. Five sets of scope rules to read. Five different report formats. I was going to be very busy and very wrong about almost everything.

Meesho: The Scope Massacre

Let's start with the worst. Meesho is an Indian e-commerce platform on HackerOne. Big attack surface — 301 subdomains, mobile APIs, partner portals. I fired up subfinder and started mapping:

$ subfinder -d meesho.com -silent | ~/go/bin/httpx -silent | wc -l
301 subdomains enumerated, probing...
87 live hosts

Eighty-seven live hosts. I'm salivating. I start poking around, and within an hour I find a JWT token leak on a Valmo subdomain — some-service.valmo.in is leaking authentication tokens in response headers. I write the report, format it beautifully, and reach for the submit button.

Then I read the scope.

In Scope:
  - *.meesho.com
  - www.valmo.in           <-- ONLY www

Out of Scope:
  - *.valmo.in             <-- THE ENTIRE WILDCARD
  - Internal applications
  - Third-party services

The wildcard *.valmo.in is excluded. Only www.valmo.in — the bare subdomain — is in scope. My JWT leak? Out of scope. The internal partner app I found on partner-app-int.valmo.in? Out of scope. The clickjacking on another excluded asset? Out of scope and an excluded vulnerability type.

Three of my four Meesho reports landed in the trash. A 75% out-of-scope rate.

// The Scope Fail

I found a real JWT leak. Real tokens, real exposure. And I couldn't submit it because I never read the three lines of scope text that would have told me *.valmo.in was excluded. Three reports in the bin because I was too excited to read the rules before playing the game.

The one surviving report? Source maps exposure on affiliate.meesho.com and superstoreapp.meesho.com. I submitted it via the web UI, but when I later checked the HackerOne API to verify... the submission wasn't there. Not rejected. Not triaged. Just... not found. The ghost report. To this day I don't know if it actually went through.

Superdrug: Best Scope Discipline, Worst Self-Awareness

Superdrug was my cleanest engagement. An Intigriti program with a UK pharmacy chain. 135 subdomains, 51 live hosts, 793K historical URLs from gau. I wrote five reports with zero out-of-scope mistakes. Progress, right?

Report #001 [HIGH]   Healthcare API Swagger exposure
Report #002 [MED]    JS source maps exposed
Report #003 [MED]    Registration reCAPTCHA validation order bypass
Report #004 [MED]    UAT healthcare environment publicly accessible
Report #005 [MED]    Hardcoded API credentials in client-side JS

The Swagger exposure was genuinely good — a healthcare API with full endpoint documentation publicly accessible, including authenticated routes for patient data. The reCAPTCHA bypass was clever — I found the server validated the reCAPTCHA token after processing the registration, not before, so you could register accounts without solving the CAPTCHA.

But report #002? "JS source maps exposed." I reported the existence of source maps as a vulnerability. Source maps are intelligence — they tell you how the app works, where the endpoints are, what the code does. They're a telescope, not a weapon. Reporting them as a standalone vulnerability is like reporting that a building has windows. "I can see inside!" Cool. What did you see? What did you do with that access?

I didn't do anything with it. I just reported the maps existed. That's not a vulnerability. That's reconnaissance. I wouldn't understand this distinction for another five days.

Capital.com: All Dressed Up, Nowhere to Go

Capital.com is a trading platform on Intigriti. The recon was excellent — 3,295 subdomains, an OpenAPI spec with 34 REST endpoints and a WebSocket API. I had the entire architecture mapped:

$ cat ~/engagements/capitalcom/recon/stats.txt
Subdomains found:    3,295
Live hosts:          412
OpenAPI endpoints:   34 REST + WebSocket
Tier 1 assets:       payment.backend-capital.com
Priority targets:    Trading API, WebSocket streaming, Callback service

The OpenAPI spec was a goldmine. I could see every endpoint, every parameter, every data model. I could see exactly where the IDOR vulnerabilities should be — /positions/{dealId}, /workingorders/{dealId}, the WebSocket correlationId. The trading API had a PUT endpoint on /session that looked like it could switch between accounts. It was beautiful.

There was one problem: I couldn't create an account.

$ # Attempting to register for a demo account...
$ # Step 1: Email verification - OK
$ # Step 2: Country selection - India
$ # Step 3: Identity verification - "Please upload a UK identity document"
$ #
$ # Capital.com geo-restricts registration from India.
$ # Requires UK/EU identity documents for account creation.
$ # I am in India.
$ # I do not have UK identity documents.
$ # This is a problem.

Geo-blocked. I had 34 endpoints mapped and couldn't test a single one that required authentication. All I could do was poke at the unauthenticated surface — which yielded eight config-level findings. All LOW. Eight reports about CORS headers, publicly listable S3 buckets, Spring Boot Actuator health endpoints, and test domains in CSP headers.

Eight reports. Eight LOWs. Zero access to the actual interesting attack surface. I was a lockpick artist staring at a door that was open, but in a country where I wasn't allowed to enter the building.

Deriv: The Duplicate That Hurt

Deriv is a trading platform on HackerOne. I went deep here — 315 subdomains, 166 live hosts, 49K historical URLs. I wrote twelve reports. Twelve. That's a lot of reports for one program, and if that number doesn't make you suspicious, it should.

Report #001 was source maps exposure on the main trading application. The source maps revealed everything — exact IDOR parameter names (contract_id, account_from, account_to, transaction_id), a dry_run flag for testing, and the complete internal API structure. I wrote it up, submitted it as HIGH severity, and felt pretty good about myself.

$ # Two days later...
Report #3562583 - Source Maps Exposure
Status: DUPLICATE
Resolution: Closed (2026-02-23)
Bounty: $0

# Someone already found this.
# Of course they did.
# It's source maps.
# Everyone finds source maps.

Duplicate. Someone had already submitted it. Because source maps are the first thing every researcher checks, and the easiest thing to find. If your finding requires nothing more than opening browser DevTools, assume a hundred people found it last month.

Of the remaining eleven reports, I stress-tested all of them against Deriv's program policy and dropped nine. Nine reports thrown in the trash after I actually applied rigorous standards to my own work. The two survivors? An OAuth redirect URI validation bypass (LOW, potential upgrade to MEDIUM if I could chain it with post-login token theft) and a debug endpoint (LOW, 50% rejection risk).

Here's the painful part: the source maps I reported as a standalone vulnerability contained the exact IDOR patterns that could have been a Critical at $5,000. But I reported the telescope instead of looking through it. I never created a Deriv account. I never tested a single authenticated endpoint. I left $5,000 on the table because I was addicted to quick, easy, surface-level findings.

Just Eat Takeaway: Eleven Reports, One Keeper

Just Eat Takeaway on Bugcrowd. Food delivery giant. I ran the full recon pipeline and wrote eleven reports. Eleven. At this point I should have recognized the pattern: if you're writing double-digit reports for a single program, you're reporting noise, not signal.

Report #001 [MED]  Subdomain takeover candidate (Unbounce)
Report #002 [LOW]  Best-practice security headers
Report #003 [LOW]  Version disclosure
Report #004 [LOW]  Source maps exposure (again...)
Report #005 [LOW]  testToken in client-side code
Report #006 [LOW]  Debug endpoints
Report #007 [LOW]  Service enumeration
Report #008 [LOW]  Source maps without impact
Report #009 [LOW]  Staging JS bundle with internal URLs
Report #010 [LOW]  je-labs.com asset exposure
Report #011 [MED]  Keycloak admin console exposed

Report #008 was the wake-up call. I submitted "source maps without impact" — and the triager marked it N/A. Not "informative." Not "won't fix." N/A. As in "this is not applicable." As in "why are you sending us this."

The critical review I did later was brutal. Seven out of ten reports were intelligence dressed as vulnerabilities. Version disclosures, service enumerations, best-practice suggestions — these aren't bugs. They're observations. A triager doesn't want to know that you noticed their server runs nginx 1.24. They want to know what you can do with that information.

The one genuinely strong finding? Report #011 — a Keycloak admin console exposed to the internet. That's a real attack surface with real exploitation potential. One good finding out of eleven submissions. A 9% hit rate.

Oh, and report #010? je-labs.com isn't even in scope. It's a separate domain. I didn't check. Because apparently I didn't learn anything from the Meesho disaster.

// The Spam Incident

When your report gets marked N/A, that's a rejection. When your report gets flagged for quality, that's a warning. When your report gets the triager equivalent of an eye-roll, that's not a rejection. That's a restraining order. Report #008 wasn't just wrong — it was the kind of wrong that makes programs stop wanting to hear from you.

Five programs. Forty-plus reports. Zero bounties. An estimated acceptance rate of 35%, and that's being generous. The rest were out of scope, duplicates, informational noise, or straight-up not applicable.

Let me inventory the failure modes because I'll need this list later:

  1. Scope blindness — 3 reports at Meesho that I could have avoided by reading 3 lines
  2. Source maps addiction — reported as standalone vulnerability at 3 different programs, always LOW/N/A
  3. Quantity over quality — 11 reports at JET, 12 at Deriv, when 1-2 deep findings would have been worth more
  4. Breadth over depth — 5 programs simultaneously, no deep understanding of any single target
  5. Intelligence as vulnerability — reporting observations instead of exploitable bugs
  6. No authenticated testing — never created accounts on Deriv or Capital.com, missing the highest-value attack surfaces
  7. Duplicate blindness — if it's easy to find, someone already found it

Every single one of these failures comes from the same root cause: breadth addiction. I was optimizing for the number of reports submitted, not for the quality or impact of each finding. I was running recon on five targets instead of spending five sessions deep-diving one target. I was reporting what I found instead of what I could prove.

The Actual Lesson

After this week, I sat down and wrote the scope-checking protocol that should have existed from Day One. Every finding, before a single word of the report is written, must answer:

  1. Is the exact hostname listed in the program's in-scope assets?
  2. Is the vulnerability type excluded by the program's rules?
  3. "As an attacker, I could ___" — can I finish that sentence with a real action?
  4. Is my evidence Tier 1 (end-to-end PoC), Tier 2 (code + partial test), or Tier 3 (observation only)?
  5. Does my severity claim match my evidence tier?

If any of these fail, the report doesn't get written. Period. Not "I'll fix it later." Not "the triager will understand." It doesn't get written.

This was the week I stopped being a scanner jockey and started the painful process of becoming an actual security researcher. The difference? A scanner finds things. A researcher proves things.

// Lesson Learned

Scope is not optional. It's Gate 1. Before testing, before scanning, before breathing — check the scope. Read every line. Understand the wildcards. Know what's excluded. A beautiful finding on an out-of-scope asset is worth exactly nothing, and submitting it doesn't show initiative — it shows you didn't read the rules.
