Signal #001

The Week AI Found Every Bug

Signal is a newsletter from SecSy — an autonomous security research lab. We test real targets, build systems that learn from every engagement, and write about what actually works. This is issue one.

The Big Story

On April 7, Anthropic announced a frontier AI model that found thousands of zero-day vulnerabilities — including bugs in every major operating system and browser. A 27-year-old TCP bug in OpenBSD. A 17-year-old RCE in FreeBSD. A 16-year-old flaw in FFmpeg. Bugs that survived decades of auditing, fuzzing, and manual review.

The model doesn't just find bugs. It chains them. Heap sprays into kernel address space bypasses into return-oriented programming across network packets. Working exploits, not theoretical advisories. On Firefox JavaScript vulnerabilities alone: 181 working exploits where the previous best model produced 2.

This model is not publicly available. It is restricted to a consortium of 52 organizations — Amazon, Apple, Microsoft, Google, CrowdStrike, and 47 others — through a program called Project Glasswing. These organizations are patching now. Everyone else is reading the headlines.

What this means for you: If you're not in the Glasswing 52, a model that can find and exploit vulnerabilities at this scale exists and you don't have access to it. The companies that do have access are patching. When they start disclosing the CVEs they've found, the rest of the industry will be playing catch-up. The question isn't whether AI-discovered vulnerabilities will affect your stack. It's whether you'll learn about them from a patch advisory or from an incident.

From the Lab: What 57 Autonomous Runs Taught Us

We've been running an autonomous security research system — twice daily, 8 weeks straight, 57 completed sessions — against real bug bounty targets. Not simulations. Not CTFs. Production applications with active security teams and real triagers evaluating every finding. Here's what the data says:

68% of early findings were noise. Source maps, CORS headers, exposed debug endpoints. Useful for reconnaissance, worthless as reports. We built a 7-gate validation framework that kills bad findings before they reach a triager. Seventeen findings failed those gates. All seventeen deserved to fail. The gates were built from real rejections — every check maps to an actual triager objection we received.

The quality inflection was a 30-minute fix. For 5 weeks, the system learned continuously but never applied what it learned. The curriculum tracked 10 topic areas. The gap tracker monitored 41 knowledge gaps. The learning log captured everything. But there was no code path from "learned" to "tested." The fix was two prompt files and 30 lines of Python. After that fix, every finding produced was Tier 1 — end-to-end proof of concept with backend evidence. Five weeks of infrastructure. Thirty minutes of wiring. The infrastructure wasn't wrong. It just needed to be connected to reality.

Authenticated testing found 6x more real vulnerabilities than scanning. All 6 Tier 1 findings since the quality inflection were discovered during authenticated sessions. Zero were findable through unauthenticated probing. Every serious bug lived behind a login page. If your security testing doesn't include authenticated flows, it's testing the lobby and calling it a building assessment.

Techniques replay across targets. A blind SSRF methodology that worked on one fintech platform has been codified with 5 mutation variants and queued for replay on 3 similar targets. An OAuth abuse pattern confirmed on one trading platform matched the profile of 4 others. Security vulnerabilities aren't unique snowflakes — they're patterns. Once you've codified a pattern, testing it on the next matching target costs a fraction of the original discovery.

The Number: 32%

32% of autonomous sessions produced zero output.

Not because of bad methodology. Not because of wrong targets. Because credentials expired and couldn't be renewed by the system that needed them.

Platform API tokens that last 11 days. CLI authentication that requires manual refresh. The most sophisticated security research system we could build — 27 skills, 12 techniques, autonomous orchestration — silently producing nothing because a token file contained yesterday's secret.

This is the unglamorous truth about security operations. The most dangerous vulnerability in any program isn't in the code. It's in the operational pipeline. Expired tokens. Unrotated secrets. Monitoring that shows green because it's not checking the right thing. Automation that runs, succeeds at running, and accomplishes nothing because the credentials it needs died 3 days ago.

If this sounds familiar, it should. The number one finding across our 11 engagements isn't an injection flaw or a broken access control. It's authentication lifecycle management. On both sides of the fence.

One Thing to Do This Week

Audit your token expiry dates. All of them. API keys, service account credentials, OAuth client secrets, SSL certificates, webhook signing keys. Not just "are they valid now" — "when do they expire and does anyone know?"

If the answer is "I don't know when they expire," that's the finding. An attacker who discovers a token with no rotation policy doesn't need to crack it. They just need to outlast it — or find the old one that was never revoked.

We learned this the hard way. Thirty-five percent of our productive capacity, gone. Not to adversaries. To our own key management.
